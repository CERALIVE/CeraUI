# RPC Communication Guide

CeraUI uses a single WebSocket connection for all device control and real-time data. There is no REST API. The backend is a Bun server; the frontend is a Svelte 5 PWA. Everything flows through oRPC over WebSocket.

## 1. Wire Format

All messages are UTF-8 JSON strings sent over a single WebSocket connection. There are two distinct envelope shapes: **oRPC call/response** and **broadcast push**.

```
ws://<device>:<port>   (default port 3001)
```

The backend accepts only one message format per direction:

| Direction | Shape | Purpose |
|-----------|-------|---------|
| Client → Server | `{ id, path, input? }` | oRPC procedure call |
| Client → Server | `{ keepalive: null }` | Client-side keepalive tick |
| Client → Server | `{ pong: true }` | Heartbeat acknowledgement |
| Server → Client | `{ id, result }` | oRPC procedure response |
| Server → Client | `{ id, error: { message, code } }` | oRPC error response |
| Server → Client | `{ [type]: data, seq?: N }` | Broadcast push |

## 2. oRPC Request / Response

### Request

```json
{
  "id": "req_42_1717430000000",
  "path": ["streaming", "start"],
  "input": { "srtla_addr": "relay.example.com", "max_br": 4000 }
}
```

- `id` — unique string generated by the client (`req_<counter>_<timestamp>`).
- `path` — array of router segments leading to the procedure (e.g. `["auth", "login"]`).
- `input` — procedure input, omitted when the procedure takes no arguments.

### Success Response

```json
{
  "id": "req_42_1717430000000",
  "result": { "success": true, "applied": { "max_br": 4000 } }
}
```

### Error Response

```json
{
  "id": "req_42_1717430000000",
  "error": {
    "message": "Unauthorized",
    "code": "INTERNAL_ERROR"
  }
}
```

The client matches responses to pending calls by `id`. Unmatched responses are logged and discarded. Calls time out after 30 seconds by default.

**Source:** `apps/frontend/src/lib/rpc/client.ts` — `call()`, `handleRPCResponse()`

## 3. Broadcast Messages

The server pushes state changes to all authenticated clients without a prior request. The envelope is:

```json
{ "<type>": <payload>, "seq": 17 }
```

- The key is the message type (e.g. `"config"`, `"sensors"`).
- `seq` is a per-type monotonic integer (see [Section 5](#5-per-type-sequence-numbers)).
- Messages sent via `sendToClient` (e.g. the post-login push) omit `seq`.

The client parses every inbound frame, lifts `seq` out of the envelope, and routes each `(type, data, seq?)` triple to `handleMessage` in `subscriptions.svelte.ts`.

**Source:** `apps/backend/src/rpc/events.ts` — `broadcast()`, `sendToClient()`; `apps/frontend/src/lib/rpc/client.ts` — `handleMessage()`

## 4. Broadcast Event Types

These are all message types handled by the `handleMessage` switch in `subscriptions.svelte.ts`. Types marked **periodic** are emitted on a fixed server-side interval; types marked **event-driven** fire when state changes.

### `config`

Streaming configuration snapshot. Emitted after any config change (start, setConfig, setBitrate) and as part of the post-login push.

```json
{
  "config": {
    "srtla_addr": "relay.example.com",
    "srt_port": 5000,
    "srt_latency": 2000,
    "max_br": 4000,
    "acodec": "aac",
    "delay": 0,
    "bitrate_overlay": false,
    "resolution": "1920x1080",
    "framerate": 30
  },
  "seq": 1
}
```

Ingestion is **lock-aware**: fields guarded by an optimistic dirty-field lock are skipped until the lock resolves. See `apps/frontend/src/lib/rpc/dirty-registry.svelte.ts`.

### `status`

Aggregated device status. Emitted on state changes and as part of the post-login push.

```json
{
  "status": {
    "is_streaming": false,
    "available_updates": null,
    "updating": null,
    "ssh": { "enabled": false },
    "wifi": { ... },
    "modems": { ... },
    "asrcs": ["default", "hw:1,0"]
  },
  "seq": 3
}
```

The `wifi` and `modems` sub-objects are also broken out into their own top-level types (see below). The `status` case in `handleMessage` merges partial updates — only fields present in the incoming payload are applied.

### `wifi`

WiFi operation result. Emitted in response to connect/disconnect/scan operations.

```json
{
  "wifi": {
    "new": { "success": true }
  }
}
```

or on auth failure:

```json
{
  "wifi": {
    "new": { "error": "auth" }
  }
}
```

The client shows a toast on `new.error === "auth"` or `new.success === true`.

### `modems`

Modem list update. Emitted periodically (~30s) and on modem state changes.

```json
{
  "modems": {
    "modem0": {
      "iface": "wwan0",
      "signal": -75,
      "operator": "Carrier",
      "state": "connected",
      "technology": "5G"
    }
  },
  "seq": 12
}
```

Coalescing window: **30 000 ms** (matches the 30s broadcast interval).

### `netif`

Network interface snapshot. Emitted periodically (~5s).

```json
{
  "netif": {
    "eth0": { "tp": 1000, "ip": "192.168.1.10", "mac": "aa:bb:cc:dd:ee:ff", "enabled": true },
    "wwan0": { "tp": 0, "ip": null, "enabled": true }
  },
  "seq": 44
}
```

Ingestion is **lock-aware** for the `enabled` field only: `BondToggle` registers per-interface locks as `enabled_<ifname>`. All other fields (`tp`, `ip`, `error`, `mac`) flow through live.

Coalescing window: **5 000 ms**.

### `sensors`

SoC telemetry. Emitted periodically (~1s).

```json
{
  "sensors": {
    "cpu_temp": 52.3,
    "cpu_load": 18.5,
    "mem_used_mb": 412,
    "mem_total_mb": 3840
  },
  "seq": 1802
}
```

Coalescing window: **1 000 ms** (matches the 1s broadcast interval). Only an exact duplicate emitted within the same 1s window is dropped; any changed value passes through immediately.

### `gateways`

Default gateway / routing table snapshot. Emitted periodically (~2s).

```json
{
  "gateways": {
    "default": "192.168.1.1",
    "interfaces": ["eth0", "wwan0"]
  },
  "seq": 901
}
```

Coalescing window: **2 000 ms**.

### `relays`

Cloud relay list. Emitted after relay config changes and as part of the post-login push (when relays are configured).

```json
{
  "relays": {
    "selected": "eu-west-1",
    "available": [
      { "id": "eu-west-1", "name": "EU West", "latency_ms": 12 }
    ]
  },
  "seq": 2
}
```

### `acodecs`

Available audio codecs. Emitted once as part of the post-login push; does not change at runtime.

```json
{
  "acodecs": {
    "aac": { "name": "AAC" },
    "opus": { "name": "Opus" }
  }
}
```

### `pipelines`

Available GStreamer pipeline definitions. Emitted once as part of the post-login push.

```json
{
  "pipelines": {
    "hardware": "rk3588",
    "pipelines": {
      "h264_hw": { "label": "H.264 Hardware", "codec": "h264" },
      "h265_hw": { "label": "H.265 Hardware", "codec": "h265" }
    }
  }
}
```

### `notifications`

Toast notifications pushed from the server. Emitted event-driven.

```json
{
  "notifications": {
    "show": [
      { "type": "warning", "msg": "Signal degraded on wwan0", "duration": 5 }
    ]
  },
  "seq": 7
}
```

`type` is one of `"error"`, `"warning"`, `"success"`, `"info"`. `duration` is in seconds; the client converts to milliseconds for the toast library.

### `ping`

Server-to-client heartbeat. Emitted every 5s by the backend heartbeat timer. The client acknowledges with `{ pong: true }` and never routes `ping` to `handleMessage` — it is consumed entirely in `client.ts`.

```json
{
  "ping": { "t": 1717430000000 },
  "seq": 360
}
```

`t` is the server's `Date.now()` at emission time. See [Section 6](#6-server-to-client-heartbeat).

### `revisions`

Software version strings. Emitted once as part of the post-login push.

```json
{
  "revisions": {
    "ceraui": "1.4.2",
    "ceracoder": "0.9.1"
  }
}
```

### `bitrate`

Hot bitrate adjustment echo. Emitted after a successful `streaming.setBitrate` call.

```json
{
  "bitrate": { "max_br": 3500 }
}
```

The client merges `max_br` into the existing `configState`.

### `log`

Log file download payload. Emitted in response to `system.getLog` / `system.getSyslog`.

```json
{
  "log": { "name": "ceraui-2026-06-03.log", "contents": "..." }
}
```

The client triggers a browser download via `downloadLog()`.

### `auth`

Authentication result. Emitted by the reconnect re-auth path (not by the server directly as a broadcast).

```json
{
  "auth": { "success": true, "auth_token": "tok_..." }
}
```

## 5. Per-Type Sequence Numbers

Every broadcast carries a top-level `seq` field. Sequence numbers are **per-type** and **monotonically increasing** from 1. They reset to 0 on server process restart.

### Server side

`apps/backend/src/rpc/events.ts` maintains a `Map<string, number>` called `seqCounters`. Before each broadcast the counter for that type is incremented and the new value is embedded in the message:

```ts
const seq = advanceSeq(seqCounters, type);
const message = JSON.stringify({ [type]: data, seq });
```

`advanceSeq` is a pure helper exported for unit testing. There is never a single global counter — each type advances independently.

### Client side

`apps/frontend/src/lib/rpc/seq-guard.ts` provides `createSeqTracker()`, which holds a `Map<string, number>` of `lastSeen` values per type. Before applying any message, `subscriptions.svelte.ts` calls:

```ts
if (seq !== undefined && seqTracker.shouldDrop(type, seq)) return;
```

`shouldDrop` returns `true` when `incomingSeq <= lastSeen` (stale or duplicate). Gaps are fine — only strictly-greater values are required, not consecutive +1 increments.

After applying the message, `seqTracker.advance(type, seq)` records the new `lastSeen`.

Messages without a `seq` field (e.g. `sendToClient` unicasts, safety-hydrate dispatches) forward `undefined` and bypass the guard entirely. This is intentional and backward-additive.

### Reset on reconnect

When the connection transitions from any state to `"connected"`, `seqTracker.resetOnReconnect()` clears all `lastSeen` entries. A server that restarted resets its counters to 0; without this reset the client would drop every message from the fresh server as stale.

**Source:** `apps/backend/src/rpc/events.ts:107-118`; `apps/frontend/src/lib/rpc/seq-guard.ts`; `apps/frontend/src/lib/rpc/subscriptions.svelte.ts:187-198, 455-458`

## 6. Server-to-Client Heartbeat

The server emits an app-level `{ ping: { t } }` message every **5 seconds** to all authenticated clients. This is the server-to-client direction only — the client already sends its own keepalive tick (see [Section 11](#11-client-to-server-keepalive)).

Bun's `ServerWebSocket` exposes a protocol-level `ping()` frame, but browsers answer those pongs at the protocol layer where JS cannot observe them. The app-level ping carries a timestamp so the client can measure staleness.

### Server emitter

`apps/backend/src/rpc/heartbeat.ts`:

```ts
export const HEARTBEAT_INTERVAL_MS = 5000;
export const HEARTBEAT_STALE_THRESHOLD_MS = 15000;

export function emitHeartbeat(now = Date.now()): void {
  broadcast("ping", { t: now });
}

export function startHeartbeat(intervalMs = HEARTBEAT_INTERVAL_MS): void { ... }
```

`startHeartbeat` is idempotent — a second call while running is a no-op.

### Client half-open detection

`apps/frontend/src/lib/rpc/heartbeat.ts` and `apps/frontend/src/lib/rpc/half-open.ts` implement the client side:

- Every inbound frame (including `ping`) calls `heartbeatTracker.recordTraffic(Date.now())`.
- The client's 10s keepalive tick also calls `shouldForceCloseHalfOpen(tracker, now, socket.readyState)`.
- If no inbound traffic has been seen for more than **15 000 ms** (`HEARTBEAT_THRESHOLD_MS`) and the socket still reports `OPEN`, the socket is force-closed.
- Force-closing triggers `onclose`, which enters the normal infinite-retry reconnect path.

The staleness threshold is strictly greater: a gap of exactly 15 000 ms is still considered alive. This makes the boundary deterministic in tests.

```
Server pings every 5s.
Client checks every 10s.
Threshold = 15s ≈ 3 missed pings.
```

When the client receives a valid `ping` frame it sends `{ pong: true }` back and returns without routing the message to `handleMessage`.

**Source:** `apps/backend/src/rpc/heartbeat.ts`; `apps/frontend/src/lib/rpc/heartbeat.ts`; `apps/frontend/src/lib/rpc/half-open.ts`; `apps/frontend/src/lib/rpc/client.ts:197-213, 237-241`

## 7. Auth / Handshake and Post-Login Push

### Initial connection

On WebSocket `open`, the backend calls `addClient(ws)` to register the socket. No state is pushed yet — the socket is unauthenticated.

If no password has been configured, the server immediately sends:

```json
{ "status": { "set_password": true } }
```

### Login

The client calls `rpc.auth.login({ password, persistent_token: true })`. This is a standard oRPC call:

```json
{ "id": "req_1_...", "path": ["auth", "login"], "input": { "password": "...", "persistent_token": true } }
```

On success the server responds:

```json
{ "id": "req_1_...", "result": { "success": true, "auth_token": "tok_..." } }
```

### Post-login push

Immediately after a successful `auth.login` response, `adapter.ts` calls `sendInitialStatusToClient(ws)`, which sends seven unicast messages to the newly authenticated socket (no `seq` — these bypass the drop-stale guard):

```
config      — current streaming config
pipelines   — available GStreamer pipelines
relays      — relay list (omitted when no relays configured)
status      — aggregated device status (is_streaming, wifi, modems, ssh, ...)
netif       — network interface snapshot
sensors     — SoC telemetry snapshot
revisions   — software version strings
acodecs     — available audio codecs
```

**Source:** `apps/backend/src/rpc/adapter.ts:82-119`

### Re-authentication on reconnect

When the socket reconnects after the operator has already authenticated this page-load, the new socket is unauthenticated. `subscriptions.svelte.ts` detects the `"connected"` transition and, if `wasAuthenticated()` is true, calls `runReconnectReauth()`.

`reauthenticateAndHydrate` in `apps/frontend/src/lib/rpc/reconnect.ts`:

1. Reads the stored credential from `localStorage["auth"]`.
2. Calls `rpc.auth.login({ password: storedToken, persistent_token: true })`.
3. Dispatches the auth result through `handleMessage("auth", result)`.
4. On success, fires a **safety hydrate**: calls `rpc.streaming.getConfig()` and `rpc.status.getStatus()` in parallel, then dispatches both results through `handleMessage`. This is defense-in-depth on top of the server's post-login push.
5. On credential rejection, clears the stored token and calls `routeToLogin()`.

The `reauthInFlight` flag prevents concurrent re-auth attempts.

**Source:** `apps/frontend/src/lib/rpc/reconnect.ts`; `apps/frontend/src/lib/rpc/subscriptions.svelte.ts:421-441, 446-468`

## 8. Command RPC Returns Applied State

Setter procedures (`streaming.start`, `streaming.setConfig`, `streaming.setBitrate`, `network.configure`, etc.) return an `applied` object containing the post-clamp, server-validated values that were actually written. The client must not assume the input values were accepted verbatim.

Example — `streaming.setBitrate`:

```json
Request:  { "path": ["streaming", "setBitrate"], "input": { "max_br": 99999 } }
Response: { "result": { "success": true, "applied": { "max_br": 8000 } } }
```

The server clamps `max_br` to the schema maximum (e.g. 8000 kbps) before writing. The `applied` value is what the server actually stored, and the subsequent `config` broadcast will echo the same clamped value.

The frontend's dirty-field registry (`dirty-registry.svelte.ts`) uses the `applied` value — not the client's intended value — as the reconciliation target. A lock is released only when the server echoes back the applied value, not when the RPC call resolves.

## 9. Server-Side Sensor Coalescing

`apps/backend/src/rpc/coalesce.ts` implements a last-value-per-type deduplication layer inside `broadcast()`. It is a plain `Map` plus a window check — no pub/sub, no middleware.

Before each broadcast, `shouldCoalesce` is called:

```ts
if (shouldCoalesce(coalesceState, type, data, now, getCoalesceWindowMs(type))) {
  return; // drop exact duplicate within window
}
updateCoalesceState(coalesceState, type, data, now);
```

A message is dropped only when **all three** conditions hold:

1. A prior value exists for this type.
2. The current time is still within `windowMs` of when that value was emitted.
3. The new value is deeply identical to the prior value (stable-stringified, sorted keys).

The window boundary is exclusive: a value landing exactly `windowMs` later is treated as a new interval and passes through.

### Coalescing windows (LOCAL profile)

| Type | Window | Matches interval |
|------|--------|-----------------|
| `netif` | 5 000 ms | 5s broadcast |
| `sensors` | 1 000 ms | 1s broadcast |
| `gateways` | 2 000 ms | 2s broadcast |
| `modems` | 30 000 ms | 30s broadcast |

Types not listed have a window of 0 ms and are never coalesced.

The observable local cadence is unchanged: the first value of each interval always passes through, and any changed value passes through immediately regardless of timing.

**Source:** `apps/backend/src/rpc/coalesce.ts`; `apps/backend/src/rpc/events.ts:141-145`

## 10. Reconnect Sequence

The transport retries forever with jittered exponential backoff. It never permanently gives up — a device reboot or transient network blip always self-heals once connectivity returns.

### Backoff formula

```
delay = min(cap≈30s, base·2^attempt) · (1 ± 30% jitter)
```

Jitter is applied on every step, not only once capped, so a fleet of clients reconnecting after the same outage spread their attempts instead of synchronising into a thundering herd.

**Source:** `apps/frontend/src/lib/rpc/backoff.ts`

### Full reconnect sequence

```
Client                                    Server
  |                                          |
  |  [socket close / half-open force-close]  |
  |                                          |
  |  handleReconnect() — jittered backoff    |
  |  attempt 1: ~1s, attempt 2: ~2s, ...     |
  |  cap: ~30s                               |
  |                                          |
  |-------- WebSocket connect() ----------->|
  |<------- onopen -------------------------| (state → "connected")
  |                                          |
  |  seqTracker.resetOnReconnect()           |  ← all lastSeen cleared
  |                                          |
  |  wasAuthenticated() === true?            |
  |  → runReconnectReauth()                  |
  |                                          |
  |-------- auth.login(storedPassword) ---->|
  |<------- { success: true } --------------|
  |                                          |
  |  dispatch("auth", { success: true })     |
  |                                          |
  |<------- post-login push (7 unicasts) ---|  ← config, pipelines, relays,
  |                                          |    status, netif, sensors,
  |                                          |    revisions, acodecs
  |                                          |
  |  safety hydrate (parallel):              |
  |-------- streaming.getConfig() -------->|
  |-------- status.getStatus() ----------->|
  |<------- config result ------------------|
  |<------- status result ------------------|
  |                                          |
  |  dispatch("config", config)              |
  |  dispatch("status", status)              |
  |                                          |
  |  [normal broadcast stream resumes]       |
  |<------- sensors (seq 1) ----------------|  ← seq reset accepted (lastSeen cleared)
  |<------- netif (seq 1) ------------------|
  |<------- ping (seq 1) -------------------|
  |  ...                                     |
```

### Half-open detection path

```
Client                                    Server
  |                                          |
  |  [TCP half-open: socket OPEN, peer gone] |
  |  [no inbound traffic for >15s]           |
  |                                          |
  |  keepalive tick fires (every 10s):       |
  |  shouldForceCloseHalfOpen() → true       |
  |  socket.close()                          |
  |                                          |
  |  onclose fires → handleReconnect()       |
  |  [continues as normal reconnect above]   |
```

### Credential expiry path

```
  |-------- auth.login(storedPassword) ---->|
  |<------- { success: false } -------------|
  |                                          |
  |  clearStoredToken()                      |
  |  routeToLogin()                          |
  |  [DisconnectedBanner shows session-      |
  |   expired state]                         |
```

## 11. Client-to-Server Keepalive

The client sends `{ keepalive: null }` every **10 seconds** while the socket is open. The server parses this silently and returns `null` from `parseMessage` — no response is sent.

This is the client-to-server direction. It is separate from the server-to-client heartbeat ping described in [Section 6](#6-server-to-client-heartbeat).

The keepalive tick also drives the half-open check: before sending the keepalive, `shouldForceCloseHalfOpen` is evaluated. If the link is stale, the socket is force-closed instead of sending the keepalive.

**Source:** `apps/backend/src/rpc/adapter.ts:39-42`; `apps/frontend/src/lib/rpc/client.ts:197-213`
